# 시나리오 룰

이 문서는 현재 게임 코드(`game.js`, `story-validator.js`, `config.js`)가 실제로 읽는 시나리오 데이터 형식을 기준으로 정리한다.

## 전체 구조

시나리오는 에피소드 단위로 관리한다.

```js
const STORY_START_EPISODE = "EP1 첫만남";

const EPISODES = {
  "EP1 첫만남": [
    { id: 1, type: "scene reset" },
    { id: 2, type: "background", name: "편의점" },
    { id: 3, type: "dialogue", speaker: "수진", text: "안녕." }
  ]
};
```

- `STORY_START_EPISODE`가 있으면 게임 시작 에피소드로 사용한다.
- 없으면 `EP1`을 먼저 찾고, 그것도 없으면 `EPISODES`의 첫 번째 에피소드로 시작한다.
- 각 에피소드 값은 노드 배열이어야 한다.
- 노드는 기본적으로 배열 순서대로 진행된다.
- 노드에 `id`가 있으면 같은 에피소드 안에서 중복되지 않는 숫자여야 한다.

## 지원 노드 타입

현재 코드에서 지원하는 `type`은 다음과 같다.

- `background`
- `character in`
- `character out`
- `clear characters`
- `clear background`
- `scene reset`
- `sound`
- `dialogue`
- `choice`
- `move`
- `endingCheck`

이 중 `background`, `character in/out`, `clear characters`, `clear background`, `scene reset`, `sound`, `move`는 명령 노드다. 화면에 멈춰서 대기하지 않고 즉시 처리된 뒤 다음 노드로 넘어간다.

`dialogue`와 `choice`는 플레이어 입력을 기다린다.

## 공통 조건

노드와 선택지는 `condition`을 가질 수 있다.

조건이 없으면 항상 실행된다. 조건이 있으면 모든 조건을 만족할 때만 실행된다.

지원 조건:

```js
condition: {
  dopamineMin: 40,
  dopamineMax: 80,
  affectionMin: 20,
  affectionMax: 70
}
```

- `dopamineMin`: 현재 도파민이 이 값 이상일 때 통과
- `dopamineMax`: 현재 도파민이 이 값 이하일 때 통과
- `affectionMin`: 현재 호감도가 이 값 이상일 때 통과
- `affectionMax`: 현재 호감도가 이 값 이하일 때 통과

조건값은 숫자여야 한다.

선택지 조건을 쓸 때는 도파민 0~100 범위 전체에서 표시 가능한 선택지가 0개가 되는 구간이 없도록 확인해야 한다. 예를 들어 `dopamineMax: 50`과 `dopamineMin: 80`만 있으면 51~79 구간에서 선택지가 사라진다.

## 효과

대사와 선택지는 `effects`를 가질 수 있다.

```js
effects: {
  dopamine: -10,
  affection: 15
}
```

- `dopamine`: 현재 도파민에 더하거나 뺀다.
- `affection`: 현재 호감도에 더하거나 뺀다.
- 두 값은 적용 뒤 0~100 범위로 제한된다.
- 대사 노드의 효과는 대사를 클릭해 넘길 때 적용된다.
- 선택지 효과는 해당 선택지를 누르는 순간 적용된다.

## 이름 치환

대사와 선택지 문구 안의 `000`, `OO`는 플레이어가 입력한 이름으로 치환된다.

```js
{ type: "dialogue", speaker: "주인공", text: "나는 000이야." }
```

주인공 이름이 `현수`라면 화면에는 `나는 현수이야.`처럼 표시된다.

## 배경 노드

배경을 바꿀 때 사용한다.

```js
{
  id: 2,
  type: "background",
  name: "편의점"
}
```

- `name`은 `config.js`의 `ASSET_MANIFEST.backgrounds` 키와 매칭된다.
- 현재는 엑셀에 있는 배경명을 임시 에셋에 매핑해 둔 상태다.
- 앞으로 실제 배경을 추가할 때는 엑셀의 배경명을 그대로 키로 등록하면 된다.

### CG 배경 원칙

`name`에 `(CG)`가 붙은 배경은 캐릭터 일러스트가 포함된 한 장면 이미지로 본다.

- CG 배경이 화면에 나오는 동안에는 개별 캐릭터 일러스트를 띄우지 않는다.
- CG 배경 직후에는 `clear characters`로 기존 캐릭터를 비운다.
- CG 배경은 다른 `background`, `clear background`, `scene reset`, 에피소드 이동이 나오기 전까지 유지한다.
- CG 대사/선택지가 끝난 뒤 일반 배경으로 돌아가야 한다면 그 시점에 필요한 `background`, `character in`을 명시적으로 다시 넣는다.
- CG 배경 바로 뒤에 일반 배경이나 `character in`이 오면 코드가 명령 노드를 연속 처리하면서 CG를 즉시 덮어버릴 수 있으므로 주의한다.

## 캐릭터 노드

캐릭터를 등장시키거나 퇴장시킬 때 사용한다.

```js
{
  id: 3,
  type: "character in",
  name: "수진",
  emotion: "Happy"
}
```

```js
{
  id: 8,
  type: "character out",
  name: "수진"
}
```

- `name`은 `config.js`의 `ASSET_MANIFEST.characters` 키와 매칭된다.
- `emotion`은 해당 캐릭터의 표정 키와 매칭된다.
- 표정 키가 없으면 `default` 이미지를 사용한다.
- 현재 엑셀의 감정명은 나중에 그대로 추가할 수 있도록 원문을 유지하는 방향이다.
- `주인공`의 `character in`은 현재 데이터에서 제거했다. 주인공 스프라이트가 없고, 주인공 등장이 화면 정렬에 방해가 되기 때문이다.

## 초기화 노드

화면 상태를 정리할 때 사용한다.

```js
{ id: 1, type: "scene reset" }
{ id: 2, type: "clear characters" }
{ id: 3, type: "clear background" }
```

- `clear characters`: 화면의 모든 캐릭터 제거
- `clear background`: 현재 배경 제거
- `scene reset`: 캐릭터와 배경을 모두 제거

## 사운드 노드

소리를 재생하거나 멈출 때 사용한다. 사운드 노드는 즉시 처리되고 다음 노드로 넘어간다.

```js
{
  id: 5,
  type: "sound",
  soundType: "bgm",
  action: "play",
  name: "nightLibrary",
  volume: 0.6
}
```

```js
{
  id: 10,
  type: "sound",
  soundType: "bgm",
  action: "stop"
}
```

```js
{
  id: 12,
  type: "sound",
  soundType: "effect",
  name: "doorOpen",
  volume: 0.8
}
```

- `soundType`은 `bgm` 또는 `effect`를 사용한다.
- `music`은 `bgm`으로, `sfx`와 `effects`는 `effect`로 처리된다.
- `action`은 `play` 또는 `stop`을 사용한다.
- `action`을 생략하면 `play`로 처리된다.
- 효과음에서 `soundType`을 생략하면 `effect`로 처리된다.
- `volume`은 0~1 사이 숫자다.
- `bgm`은 기본적으로 반복 재생된다. `loop: false`를 넣으면 한 번만 재생된다.
- 실제 사운드 파일은 `config.js`의 `ASSET_MANIFEST.sounds`에 등록해야 한다.

## 대사 노드

대사를 표시하는 노드다.

```js
{
  id: 4,
  type: "dialogue",
  speaker: "수진",
  text: "안녕, OO아.",
  effects: {
    affection: 5
  }
}
```

- `speaker`와 `text`가 필요하다.
- `speaker`가 `주인공`이면 플레이어 이름으로 표시된다.
- `speaker`가 `END`인 대사는 엔딩 화면으로 넘어가며, `text`가 엔딩 문구로 사용된다.
- `effects`가 있으면 해당 대사를 클릭해 넘길 때 적용된다.

## 선택지 노드

선택지를 표시하는 노드다.

```js
{
  id: 10,
  type: "choice",
  prompt: "뭐라고 말할까?",
  choices: [
    {
      text: "같이 가자.",
      effects: { affection: 10 },
      nextNode: 12
    },
    {
      text: "다음에 보자.",
      nextNode: 15
    }
  ]
}
```

- `prompt`는 선택지 창의 질문 문구다.
- `choices`는 하나 이상의 선택지 배열이어야 한다.
- 각 선택지는 `text`와 `nextNode`를 가져야 한다.
- 선택지에서 다른 에피소드로 직접 이동하지 않는다.
- 선택지의 `nextNode`는 같은 에피소드 안의 노드 `id`를 가리킨다.
- 다른 에피소드로 이동해야 한다면 선택지의 `nextNode`가 `move` 노드를 가리키게 만든다.
- 과거 방식인 선택지의 `next`도 코드에 남아 있지만, 현재 룰에서는 사용하지 않는 것을 권장한다.

## follow 대사

선택지 직후에만 나오는 추가 대사는 선택지 안의 `follow`에 넣는다.

```js
{
  text: "괜찮아?",
  effects: { affection: 5 },
  follow: [
    { speaker: "주인공", text: "응, 괜찮아." },
    { speaker: "수진", text: "다행이다.", effects: { affection: 3 } }
  ],
  nextNode: 12
}
```

- `follow`는 배열이다.
- 각 follow 행은 대사처럼 `speaker`, `text`, `effects`를 가질 수 있다.
- 선택지를 누르면 선택지 효과가 먼저 적용된다.
- 그 다음 follow 대사가 순서대로 표시된다.
- follow 대사가 모두 끝나면 `nextNode`로 이동한다.
- 엑셀 변환에서는 follow 행을 별도 표시한 뒤 이 구조로 넣는다.

## 이동 노드

에피소드나 같은 에피소드의 특정 노드로 이동할 때 사용한다.

```js
{
  id: 20,
  type: "move",
  next: "EP3 동아리 MT"
}
```

```js
{
  id: 21,
  type: "move",
  nextNode: 30
}
```

```js
{
  id: 22,
  type: "move",
  next: "EP4-A 넷이",
  nextNode: 5
}
```

- `next`가 있으면 해당 에피소드로 이동한다.
- `nextNode`가 있으면 대상 에피소드 안의 해당 `id` 노드로 이동한다.
- `next` 없이 `nextNode`만 있으면 현재 에피소드 안에서 이동한다.
- `next`와 `nextNode`가 둘 다 없으면 검증 경고 대상이다.
- 에피소드가 끝나며 다른 에피소드나 미니게임으로 넘어갈 때는 `move` 직전에 `clear background`를 실행해야 한다.
- 선택지가 마지막 `move` 노드의 `id`를 직접 가리키는 경우가 있으므로, 기존 `move id` 자리에 `clear background`를 두고 바로 다음 노드에 실제 `move`를 배치하는 방식이 안전하다.

```js
{
  id: 24,
  type: "clear background"
},
{
  type: "move",
  next: "MINIGAME",
  after: "EP2 동아리 OT"
}
```

## 미니게임 이동

미니게임은 `move` 노드에서만 실행하는 것을 원칙으로 한다.

```js
{
  id: 30,
  type: "move",
  next: "MINIGAME",
  minigame: "brickBreaker",
  after: "EP3 동아리 MT",
  afterNode: 40,
  options: {
    maxTurns: 10
  }
}
```

- `next: "MINIGAME"`이면 도파민 준비 씬으로 이동한다.
- `minigame` 또는 `subGame`에 실행할 미니게임 ID를 넣는다.
- 생략하면 기본값은 `brickBreaker`다.
- 지원 미니게임 ID는 `brickBreaker`, `sideShooter`다.
- `after`는 미니게임 종료 후 돌아갈 에피소드다.
- `afterNode`는 돌아갈 에피소드 안의 노드 `id`다.
- `after`가 없으면 기본값으로 `EP_AFTER_MINIGAME`을 찾는다.
- `options`는 미니게임 생성자 두 번째 인자로 전달되는 설정이다.
- `brickBreaker`는 `options.maxTurns`로 진행 턴 수를 조절한다. 값이 클수록 더 오래 버텨야 한다.
- `sideShooter`는 `options.difficulty`와 `options.durationSeconds`를 받는다. `difficulty`가 클수록 적 체력이 강해진다.

## 엔딩 체크

자동 엔딩 분기를 실행하려면 `endingCheck` 노드를 사용한다.

```js
{
  id: 50,
  type: "endingCheck"
}
```

현재 코드의 엔딩 판정은 다음 순서다.

- 도파민이 40 미만이면 `low`
- 도파민이 70 초과이면 `high`
- 호감도가 40 미만이면 `bad`
- 그 외에는 `good`

단순히 특정 엔딩 문구를 바로 보여주고 싶으면 `speaker: "END"`인 대사를 사용한다.

```js
{
  type: "clear background"
},
{
  id: 99,
  type: "dialogue",
  speaker: "END",
  text: "TRUE END"
}
```

## 검증 기준

`story-validator.js`는 다음 항목을 검사한다.

- 에피소드 값이 배열인지
- 노드 `id`가 숫자이고 중복되지 않는지
- 지원하지 않는 `type`이 있는지
- 조건 키와 조건값 형식이 올바른지
- 배경 노드에 `name`이 있는지
- 캐릭터 등장 노드에 `name`이 있는지
- 사운드 타입, 동작, 볼륨이 올바른지
- `move`의 이동 대상이 존재하는지
- 선택지에 `text`와 `nextNode`가 있는지
- 선택지의 `nextNode` 대상이 존재하는지
- follow가 배열이고 각 행에 `speaker`, `text`가 있는지

검증 함수는 게임 구동 필수 로직이 아니라 데이터 점검용이다.

## 엑셀 변환 관련 메모

- 변수명과 조건명은 엑셀 기준을 우선한다.
- 엑셀의 배경명, 사운드명, 감정명은 가능한 원문 그대로 데이터에 넣는다.
- 실제 에셋 등록은 `config.js`에서 같은 이름의 키를 추가하는 방식으로 맞춘다.
- 엑셀에서 `HighDo`처럼 높은 도파민 선택지를 뜻하지만 중간 구간 선택지가 따로 없는 경우, 현재 데이터에서는 51~100 구간을 커버하도록 보정했다.
- 명시적으로 `HIGH (80-100)`처럼 범위가 적힌 경우에는 그 범위를 유지한다.
