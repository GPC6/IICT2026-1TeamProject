# 도파민 벽돌깨기 수정요청 구현계획

작성 기준:

- `sub_games/벽돌깨기_현재_게임디자인.md`의 `요청:` 항목
- `sub_games/breakblocks.js`의 현재 구현

목표는 현재 벽돌깨기를 “도파민을 0~100 안에서 관리하며, 일부러 실패하기보다 끝까지 조절하도록 유도하는 미니게임”으로 정리하는 것이다.

## 1. 요청사항 목록

현재 문서에서 확인한 요청은 다음 8개다.

| 번호 | 요청 | 관련 영역 |
| --- | --- | --- |
| R1 | 게임 내부 도파민도 `0~100`으로 제한 | 도파민 모델 |
| R2 | 초기화 시 턴 수를 정할 수 있게 하기 | 설정/테스트 |
| R3 | 가장 첫 공이 바닥에 닿은 위치를 다음 턴 발사 기준점으로 사용 | 공 발사/턴 구조 |
| R4 | 공 개수 간격을 도파민 5씩으로 설정 | 공 개수 규칙 |
| R5 | 생성확률, 도파민 증가/감소량, 내구도를 계산해 평균적으로 `50~80` 도파민을 얻도록 조절 | 밸런싱 |
| R6 | 블럭 사이에 아이템 생성, 효과가 시각적으로 바로 보이게 하기, 아이템 종류 추가 고려 | 아이템/시각 피드백 |
| R7 | 점수제 제거 | UI/상태 |
| R8 | 블럭이 바닥선에 닿으면 결과가 좋지 않게 하기 | 실패 패널티/악용 방지 |

## 2. 구현 우선순위 요약

추천 순서는 다음과 같다.

1. 도파민 범위와 점수 제거
2. 턴 수 설정과 테스트 진입점 연동
3. 발사 기준점 이동
4. 공 개수 규칙 변경
5. 블럭/도파민 밸런싱
6. 바닥선 실패 패널티
7. 아이템 시스템 추가
8. 플레이 테스트와 재조정

이 순서를 추천하는 이유는 앞 단계가 뒤 단계의 계산 기준이 되기 때문이다. 특히 R1, R4, R5는 서로 강하게 연결되어 있어서 도파민 상한과 공 개수 규칙을 먼저 고정해야 평균 도파민 밸런스를 계산할 수 있다.

## 3. 1차 작업: 핵심 상태 정리

### 대상 요청

- R1: 게임 내부 도파민도 `0~100`
- R7: 점수제 제거

### 구현 방향

현재 미니게임 내부 도파민은 `0~120`까지 허용된다. 이를 전부 `0~100`으로 바꾼다.

수정 대상:

- `hitBrick()`
- `drawMeter()`
- `getDopamine()`
- 도파민 과부하 판정
- 미터 색상 구간

점수는 미니게임의 의사결정에 기여하지 않으므로 제거한다.

제거 대상:

- `this.score`
- 블럭 타격 점수 증가
- 회복 블럭 파괴 보너스
- 헤더의 점수 표시
- 종료 화면의 최종 점수 표시

### 성공 기준

- 게임 내부 모든 도파민 계산이 `0~100` 안에 머문다.
- 화면에 점수 관련 텍스트가 나오지 않는다.
- 본게임 복귀 시 기존처럼 `getDopamine()`으로 도파민만 반환한다.

## 4. 2차 작업: 턴 수 설정

### 대상 요청

- R2: 초기화 시 턴 수를 정할 수 있게 하기

### 구현 방향

`BrickBreakerGame` 생성자에 선택 옵션을 추가한다.

예상 형태:

```js
new BrickBreakerGame(initialDopamine, { maxTurns: 10 })
```

기본값은 기존과 동일하게 `10`으로 둔다.

테스트 페이지에서는 초기 도파민 입력 옆에 턴 수 입력을 추가할 수 있다. 본게임에서 호출할 때는 옵션을 넘기지 않으므로 기존 동작을 유지한다.

### 성공 기준

- 옵션이 없으면 10턴으로 진행된다.
- 테스트 페이지에서 턴 수를 바꾸면 해당 턴 수 기준으로 종료된다.
- 본게임 미니게임 진입 흐름은 깨지지 않는다.

## 5. 3차 작업: 다음 턴 발사 기준점

### 대상 요청

- R3: 가장 첫 공이 바닥에 닿은 위치를 다음 턴 기준점으로 사용

### 구현 방향

현재 발사 기준점은 매번 고정이다.

```js
this.w / 2, this.floor - 18
```

이를 상태값으로 분리한다.

예상 상태:

```js
this.launchX = this.w / 2;
this.launchY = this.floor - 18;
this.nextLaunchX = null;
```

공 중에서 가장 먼저 바닥에 닿은 공의 `x` 좌표를 저장한다.

주의할 점:

- 첫 공이 좌우 벽 근처에 떨어져도 다음 발사가 불가능하지 않도록 `x`를 안전 범위로 제한한다.
- 한 턴에 여러 공이 떨어져도 최초 1회만 기준점으로 저장한다.
- 조준선과 공 시작 위치 모두 이 기준점을 사용해야 한다.

### 성공 기준

- 첫 턴은 중앙에서 시작한다.
- 두 번째 턴부터는 직전 턴에서 가장 먼저 떨어진 공 위치에서 시작한다.
- 기준점이 좌우 벽 밖으로 나가지 않는다.

## 6. 4차 작업: 공 개수 규칙 변경

### 대상 요청

- R4: 공 개수 간격을 도파민 5씩으로 설정

### 구현 방향

현재 공 개수는 도파민 `0~100`을 `1~9`개로 매핑한다.

요청대로 도파민 5 단위로 공 개수를 계산한다.

추천 규칙:

```text
공 개수 = max(1, floor(dopamine / 5))
```

예시:

| 도파민 | 공 개수 |
| --- | --- |
| 0~4 | 1 |
| 5~9 | 1 |
| 10~14 | 2 |
| 50~54 | 10 |
| 80~84 | 16 |
| 100 | 20 |

단, 공이 최대 20개까지 늘어나면 현재 블럭 수/속도/턴 길이에 영향이 크다. 따라서 구현 시 아래 중 하나를 선택해야 한다.

| 방식 | 장점 | 단점 |
| --- | --- | --- |
| `floor(dopamine / 5)` 그대로 사용 | 요청을 가장 직접 반영 | 고도파민에서 공이 너무 많아질 수 있음 |
| `floor(dopamine / 5)` 후 상한 설정 | 게임 속도 안정 | 요청의 의도가 조금 약해짐 |

현재 요청 문맥상 5 단위 체감을 주는 것이 중요하므로, 우선 상한 없이 구현한 뒤 플레이 테스트에서 필요하면 상한을 추가하는 편이 좋다.

### 성공 기준

- 도파민 5 단위로 공 개수가 바뀐다.
- 도파민 0이어도 최소 1개는 발사된다.
- 도파민 100에서 공 개수가 20개가 된다.

## 7. 5차 작업: 블럭 생성과 도파민 밸런싱

### 대상 요청

- R5: 생성확률, 도파민 증가/감소량, 내구도를 계산해 평균적으로 `50~80` 도파민을 얻도록 조절

### 구현 방향

이 단계는 앞의 R1, R4가 반영된 뒤 조정해야 한다.

목표는 플레이어가 무작위로 쏘더라도 극단값으로 너무 쉽게 치우치지 않고, 의식적으로 회복 블럭을 노리면 `50~80` 구간에 머물 수 있게 하는 것이다.

현재 문제:

- 자극 블럭 비율이 높다.
- 자극 블럭은 타격마다 즉시 도파민을 올린다.
- 회복 블럭은 파괴되어야만 도파민을 낮춘다.
- 도파민이 높을수록 공이 많아져 자극 블럭을 더 많이 맞히기 쉽다.

추천 조정안:

| 항목 | 현재 | 1차 조정안 |
| --- | --- | --- |
| 칸당 블럭 생성률 | 66% | 55~60% |
| 자극/회복 비율 | 72% / 28% | 60% / 40% |
| 자극 블럭 도파민 | `+1~+3` 타격당 | `+1~+2` 타격당 |
| 회복 블럭 도파민 | `-10~-25` 파괴 시 | `-8~-18` 파괴 시 |
| 자극 블럭 체력 | `1~3` | `1~2` |
| 회복 블럭 체력 | `2~4` | `1~3` |

이 조정안은 회복 블럭을 조금 더 자주, 조금 더 쉽게 터뜨리게 해서 고도파민 상태에서 회복 선택지를 실제 선택지로 만든다.

### 검증 방법

수동 플레이만으로는 평균을 잡기 어렵다. 구현 후 간단한 시뮬레이션 함수를 임시로 돌려 아래를 확인하는 것을 추천한다.

- 초기 도파민 50에서 10턴 종료 평균
- 초기 도파민 30에서 10턴 종료 평균
- 초기 도파민 80에서 10턴 종료 평균
- 바닥 실패를 제외한 정상 종료 시 최종 도파민 분포

### 성공 기준

- 정상 종료 결과의 상당수가 `50~80`에 분포한다.
- 회복 블럭을 의도적으로 노리면 고도파민을 낮출 수 있다.
- 아무렇게나 쏘면 항상 좋은 결과가 나오지는 않는다.

## 8. 6차 작업: 바닥선 실패 패널티

### 대상 요청

- R8: 블럭이 바닥선에 닿으면 결과가 좋지 않게 하기

### 구현 방향

현재 바닥선 실패는 종료 메시지만 다르고, 최종 도파민은 그 시점의 값을 그대로 반환한다. 이러면 플레이어가 원하는 도파민 값에서 일부러 실패하는 악용이 가능하다.

추천 규칙:

```text
블럭 바닥 실패 시 최종 도파민을 불안정 상태로 강제한다.
```

구체안:

| 방식 | 설명 |
| --- | --- |
| 강제 과부하 | 도파민을 `100`으로 설정 |
| 강제 결핍 | 도파민을 `0`으로 설정 |
| 현재 상태 기준 악화 | 현재 도파민이 50 이상이면 `100`, 50 미만이면 `0` |

추천은 `현재 상태 기준 악화`다. 일부러 실패해도 안정 구간으로 끝낼 수 없고, 현재 플레이 상태가 나쁘게 증폭된다는 의미가 분명하다.

### 성공 기준

- 바닥선 실패 시 최종 도파민이 안정 구간으로 유지되지 않는다.
- 종료 화면에서 실패 이유와 악화된 결과가 명확히 보인다.
- 정상 10턴 생존 종료와 실패 종료가 결과 면에서 구분된다.

## 9. 7차 작업: 아이템 시스템

### 대상 요청

- R6: 블럭 사이 아이템 생성, 효과 시각화, 아이템 종류 추가

### 구현 방향

현재 랜덤 턴 버프를 실제 필드 아이템으로 바꾸거나, 최소한 필드 아이템과 턴 버프를 함께 쓰는 구조로 바꾼다.

추천은 턴 시작 랜덤 버프를 제거하고, 공이 필드 아이템에 닿았을 때 즉시 효과가 발동되도록 만드는 것이다. 그래야 플레이어가 아이템을 노리는 의미가 생긴다.

### 추천 아이템

| 아이템 | 효과 | 시각 피드백 |
| --- | --- | --- |
| 완화 큐브 | 일정 시간 또는 이번 턴 자극 증가량 감소 | 공 색을 민트색으로 변경 |
| 고자극 큐브 | 일정 시간 자극 증가량 증가 | 공 색을 분홍/노랑으로 변경 |
| 강화 큐브 | 일정 시간 블럭 피해량 증가 | 공 주변에 빛나는 테두리 |
| 안정화 큐브 | 도파민을 60 근처로 조금 끌어당김 | 도파민 바에 초록 파동 |
| 분산 큐브 | 다음 발사 공 개수 일부 감소 대신 회복 효과 강화 | 발사점 주변 푸른 링 |

### 생성 위치

아이템은 블럭과 같은 격자 기준으로 생성하되, 블럭이 없는 칸에만 생성하는 것이 안전하다.

예상 규칙:

```text
새 줄 생성 시 빈 칸 중 낮은 확률로 아이템 생성
```

### 구현 주의점

- 아이템과 블럭 충돌 판정을 분리한다.
- 아이템은 공이 닿으면 즉시 제거한다.
- 현재 활성 효과를 HUD에 짧게 표시한다.
- 효과가 적용된 동안 공 색상이나 블럭 색상이 바뀌어야 한다.

### 성공 기준

- 블럭 사이 빈 공간에 아이템이 보인다.
- 공이 아이템을 먹으면 즉시 효과가 난다.
- 플레이어가 어떤 효과가 발동 중인지 화면에서 알 수 있다.
- 기존 랜덤 턴 버프보다 선택성이 높아진다.

## 10. 8차 작업: 플레이 테스트와 조정

### 대상 요청

- R5 전체
- R6 이후 체감 조정

### 구현 방향

미니게임 테스트 진입점에서 최소 아래 조합을 확인한다.

| 초기 도파민 | 턴 수 | 확인할 것 |
| --- | --- | --- |
| 30 | 10 | 낮은 도파민에서 회복 가능한지 |
| 50 | 10 | 안정 구간 유지가 가능한지 |
| 80 | 10 | 과부하를 낮출 수 있는지 |
| 50 | 5 | 짧은 테스트에서도 결과가 너무 흔들리지 않는지 |

### 성공 기준

- 정상 플레이에서 최종 도파민이 대체로 `50~80`을 노릴 수 있다.
- 일부러 바닥 실패를 노리는 것이 이득이 아니다.
- 아이템이 “운빨 보너스”가 아니라 조준 목표로 작동한다.

## 12. 우선순위 판단

가장 먼저 해야 할 것은 R1과 R7이다.

도파민 범위와 점수 제거는 다른 요청의 기준이 되는 작업이다. 이 기준이 정리되지 않으면 공 개수, 블럭 수치, 실패 패널티의 밸런스가 계속 다시 바뀐다.

그다음은 R2, R3, R4다.

이 세 요청은 플레이의 기본 구조를 바꾼다. 특히 첫 공 낙하지점 발사는 전략성을 크게 바꾸고, 도파민 5 단위 공 개수는 난이도와 블럭 수치에 직접 영향을 준다.

R5와 R8은 그 다음이다.

기본 구조가 확정된 뒤 블럭 생성과 실패 결과를 조절해야 계산이 맞다.

R6 아이템 시스템은 가장 마지막 큰 기능으로 두는 것이 좋다.

아이템은 재미와 가시성을 크게 올리지만, 블럭/공/도파민 기본 규칙이 고정되기 전에 넣으면 밸런스 기준이 흔들린다.
